# Cocos Creator 开发规则
## 基本信息
- 引擎版本：Cocos Creator 3.8+（如果是2.x就改这里）
- 语言：TypeScript
- 编辑器：VS Code + Cocos Creator 编辑器
- 核心工作：修复 Bug、新增功能、少量跨文件调整

## 目录约定（按你实际项目改）
- 业务脚本：`src/components/` `src/game/` `src/ui/`
- 工具/通用：`src/utils/` `src/common/`
- 类型/常量：`src/types/` `src/constants/`
- 资源/配置：`assets/scripts/` `assets/config/`
- ❗ 非指定目录不要擅自修改

## 命名规范
- 组件类：`XxxComp`（如 `SkillComp`）
- 接口：`IXxx`（如 `ISkillConfig`）
- 枚举：`EXxx`（如 `ESkillType`）
- 变量/函数：小驼峰；常量：全大写下划线
- 避免与 Cocos 内置 API 同名

## 修 Bug 专用规则（重点）
1. 先分析原因：问题现象 → 可能原因 → 影响范围，再动手
2. 最小修改原则：只改出错逻辑，不重构无关代码、不换结构
3. 保留原逻辑意图：别改对的流程，只修异常分支
4. 改完补充说明：改了什么、为什么改、怎么验证
5. 不要随意改生命周期顺序、节点层级、事件监听方式

## 新增功能专用规则（重点）
1. 先对齐现有结构：复用已有组件/工具/类型，不重复造轮子
2. 新增先声明：需要加类型/接口/常量时，先在对应文件里定义
3. 边界处理要全：判空、默认值、异常捕获、日志提示
4. 不破坏旧功能：新逻辑走新分支，老流程保持不变
5. 涉及跨文件：先列“影响文件清单”和“改动点说明”，确认后再执行

## 跨文件修改通用规则
1. 范围必须明确：只动指定目录/文件，不扫整个项目
2. 先列清单再改：列出所有文件 + 每个文件的具体改动
3. 分批执行：文件多就分模块/分组改，改完一组核对一组
4. 保持一致性：命名、格式、注释与现有代码对齐
5. 引用别乱删：确认无调用再删，避免连锁报错

## 禁止事项
- 不编造不存在的 Cocos API、节点属性、生命周期
- 不擅自改 `@ccproperty`、节点路径、资源引用
- 不随意删除注释、现有日志、边界判断
- 不确定的地方：先问，别猜，别硬写
